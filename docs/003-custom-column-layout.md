# Custom Column Layout Design

## Status

Completed

## Background

現在の ghostty-pane-splitter は均一グリッド（NxM）のみをサポートしている。すべての列が同じ行数を持つため、「左側に 1 pane、右側に 3 pane」のような非対称レイアウトを作成できない。

```
# 現在のツールで作成可能なレイアウト（均一グリッド）
+-------+-------+       +-------+-------+-------+
|       |       |       |       |       |       |
+-------+-------+       +-------+-------+-------+
|       |       |       |       |       |       |
+-------+-------+       +-------+-------+-------+
    2x2                         3x2

# 現在作成できないレイアウト（非対称）
+-------+-------+       +-------+---+---+
|       |       |       |       |   |   |
|       +-------+       |       +---+---+
|       |       |       |       |   |   |
|       +-------+       |       +---+---+
|       |       |       |       |   |   |
+-------+-------+       +-------+---+---+
  1列目: 1行               1列目: 1行
  2列目: 3行               2列目: 3行 x 2列 (ネスト → 非対応)
```

実際のターミナル作業では、エディタを左に大きく表示し、右側に複数の小さな pane（ターミナル、ログ、テスト実行など）を配置するユースケースが一般的である。このような非対称レイアウトをサポートすることで、ツールの実用性が大幅に向上する。

## Goals

- 列ごとに異なる行数を指定できるカスタムレイアウトをサポートする
- 既存の均一グリッド指定（`4`, `2x3`）との後方互換性を維持する
- シンプルで直感的な CLI 構文を提供する

## Non-Goals

- 任意にネストされたレイアウト（tmux のペインツリーのような構造）
- 行方向の非対称レイアウト（例: 上段 1 pane、下段に横 3 pane）
  - 将来の拡張として検討可能だが、本設計の初期スコープには含めない
- pane のサイズ比率の指定（Ghostty の equalize で均等化のみ対応）

## Design

### CLI 構文

カンマ区切りで各列の行数を指定する:

```
ghostty-pane-splitter <col1_rows>,<col2_rows>[,<col3_rows>,...]
```

#### 使用例

```bash
# 左1 pane、右3 pane
ghostty-pane-splitter 1,3

# 左2 pane、中央1 pane、右3 pane
ghostty-pane-splitter 2,1,3

# 3列、各2行（2x3 と等価）→ 既存のグリッド指定でも可
ghostty-pane-splitter 2,2,2
```

視覚的な対応:

```
ghostty-pane-splitter 1,3        ghostty-pane-splitter 2,1,3

+----------+----------+          +------+------+------+
|          |          |          |      |      |      |
|          +----------+          +------+      +------+
|          |          |          |      |      |      |
|          +----------+          +------+      +------+
|          |          |          |      |      |      |
+----------+----------+          +------+------+------+
```

#### 構文の判定ルール

入力に対して以下の順序でパースを試みる:

1. カンマを含む → カスタムレイアウト（例: `1,3`）
2. `x` を含む → グリッド指定（例: `2x3`）— 既存動作
3. 数値のみ → 数値指定（例: `4`）— 既存動作

**注意: 数値指定とカスタム指定の違い**

カンマなしの数値 `3` は既存の数値指定として解釈され、因数分解により `[3, 1]`（3列1行）になる。1列3行としたい場合はグリッド指定 `1x3` またはカスタム指定 `3,` ではなく `1,1,1`（3列各1行）のように指定する必要がある。`3,` のような末尾カンマや `,3` のような先頭カンマは不正な入力としてエラーにする。

### データモデルの変更

現在の `Layout` 構造体:

```rust
pub struct Layout {
    pub cols: u32,
    pub rows: u32,
}
```

新しい `Layout` 構造体:

```rust
pub struct Layout {
    /// Number of rows in each column. columns.len() = number of columns.
    pub columns: Vec<u32>,
}
```

均一グリッドもこのモデルで表現可能:

| 入力 | `columns` | 意味 |
|------|-----------|------|
| `4` | `[2, 2]` | 2x2 均一グリッド |
| `2x3` | `[3, 3]` | 2 列 x 3 行の均一グリッド |
| `1,3` | `[1, 3]` | 左 1 行、右 3 行 |
| `2,1,3` | `[2, 1, 3]` | 3 列で各 2, 1, 3 行 |

ヘルパーメソッドを追加:

```rust
impl Layout {
    /// Total number of columns.
    pub fn num_cols(&self) -> usize {
        self.columns.len()
    }

    /// Total number of panes.
    pub fn total_panes(&self) -> u32 {
        self.columns.iter().sum()
    }
}
```

### バリデーション

- 各列の行数は 1 以上
- 列数は 1 以上
- 総 pane 数は 2 以上（1 pane だと分割の意味がない）
- カンマ区切りの各要素が有効な正整数であること
- 先頭・末尾のカンマや連続カンマは不正入力としてエラー（例: `,3`, `3,`, `1,,3`）

### 分割アルゴリズムの変更

現在のアルゴリズムは「全列で同じ行数を split_down する」前提だが、新しいアルゴリズムは列ごとに異なる行数で split_down する。変更は最小限で済む。

#### Ghostty の pane 移動順序の前提

Ghostty の `goto_split:next` / `goto_split:previous` は **pane の作成順** でフォーカスを移動する。Phase 1 で列を左から右に作成し、Phase 3 で各列内を上から下に分割するため、`goto_next` は常に「次の列の先頭 pane」に移動する。この前提は既存の均一グリッド実装（`split.rs`）と同じである。

#### アルゴリズム

```
Phase 1: Create columns
  split_right を (num_cols - 1) 回実行

Phase 2: Navigate back to first column
  goto_previous を (num_cols - 1) 回実行

Phase 3: Create rows in each column
  for each column i:
    split_down を (columns[i] - 1) 回実行
    if i < num_cols - 1:
      goto_next を 1 回実行

Phase 4: Equalize pane sizes
  equalize_splits を 1 回実行

Phase 5: Navigate to top-left pane
  goto_previous を (total_panes - 1) 回実行
```

既存の均一グリッドのフローと比較した差分:
- Phase 3: `split_down` 回数が列ごとに可変
- Phase 5: 処理終了後に最も左上の pane（最初に作成された pane）にフォーカスを戻す。`goto_previous` を総 pane 数 - 1 回実行することで、作成順の先頭 pane に確実に移動する

### 出力メッセージの変更

均一グリッドとカスタムレイアウトで表示を分ける:

```
# 均一グリッド（全列の行数が同じ場合）
Splitting into 2x3 grid (6 panes)...

# カスタムレイアウト
Splitting into custom layout 1,3 (4 panes)...
```

### 影響範囲

| ファイル | 変更内容 |
|----------|----------|
| `src/layout.rs` | `Layout` 構造体変更、パーサー拡張 |
| `src/split.rs` | 列ごとに可変の行数で分割するように変更 |
| `src/main.rs` | 出力メッセージの変更、ヘルプテキスト更新 |
| `src/config.rs` | 変更なし |
| `src/keybind.rs` | 変更なし |

## Alternatives Considered

### ネスト記法（例: `{1|{3}}`, `1:{3}`）

任意のネスト構造を許す記法も検討したが、以下の理由から見送った:

- Ghostty のキーボードショートカットによる split は「右に分割」「下に分割」の 2 方向のみで、任意のネストの実現は操作が複雑になる
- パースが複雑になり、ユーザーにとっても記法が直感的でなくなる
- 実際のユースケースのほとんどは「列ごとに異なる行数」でカバーできる

### 行方向の非対称レイアウト

```
# 例: 上段 1 pane、下段 2 pane
+-------------------+
|                   |
+---------+---------+
|         |         |
+---------+---------+
```

これは列方向の非対称レイアウトと対称的な機能だが、ユースケースが限定的であるため初期スコープからは除外した。将来的には `--direction=row` のようなオプションや別の記法（例: `1/2` でスラッシュを行区切りにする）で対応可能。

### JSON/YAML 設定ファイル

```json
{ "columns": [{"rows": 1}, {"rows": 3}] }
```

CLI ツールとしての手軽さを重視し、コマンドライン引数のみで完結する設計を優先した。

## Testing Strategy

- **パーサーのテスト**: カスタムレイアウト構文のパースを parameterized test で網羅
  - 正常系: `1,3`, `2,1,3`, `1,1` など
  - 異常系: `0,3`, `1,0`, `1`, `,3`, `1,`, `a,b` など
  - 均一グリッドとの等価性: `2,2` と `2x2` が同じ Layout を生成すること
- **既存テストの更新**: `Layout` 構造体の変更に伴い、既存の数値指定・グリッド指定のテストを更新
- **分割実行のテスト**: 実際の Ghostty 上での手動テスト（既存方針通り）
